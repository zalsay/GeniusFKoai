## OTP 子链（PHONE_CONFIRMATION_REQUIRED 兜底）

PayPal 风控对 guest SignUp 强制电话验证。协议层完整实现 4 步子链：

1. **detect** — paypal_signup 的首次 SignUp 响应里出现
   `errors[0].message == "PHONE_CONFIRMATION_REQUIRED"`
2. **initiate** — POST `/graphql?InitiateRiskBasedTwoFactorPhoneConfirmationMutation`
   - body: {locale, phoneCountry, phoneNumber: 本地号 (剥 +1), token: ec_token}
   - 响应给 authId / challengeId / state="PENDING"
3. **poll relay_url** — 复用 `payment._fetch_ctf_relay_code` 从中转邮箱拉 6 位 PIN
4. **confirm** — POST `/graphql?ConfirmRiskBasedTwoFactorPhoneConfirmationMutation`
   - body: {authId, challengeId, pin, token}
   - 响应 state="CONFIRMED" 即通过
5. **重发 SignUp** — 同 body，PayPal 这次返回 ISSUER_DECLINE + accessToken (即 euat)

### sms_pool 轮换

stage 在重发 SignUp 收到 `OAS_ERROR` (createMemberAccount 风控) 时
自动切换 `state.sms_pool[i+1]` 重试，直到拿到 token / 池子用尽 / 出现非 OAS 错误。

## 残留问题

PayPal **OAS_ERROR**（createMemberAccount 风控）：当 sms_pool 中的号码已被
PayPal 标记（同号反复尝试 / 同 BIN 段 / 同源 IP），重发 SignUp 即使 OTP 通过
也会返回 OAS_ERROR。这**不是协议层缺陷**，是号码池资源耗尽：

- 实测 4 条号 (5822057xxx / 5822064xxx，同源 mail-api.yuecheng.shop) 全 OAS
- 每次 OTP 失败的号码会进入风控冷却期（数小时-数天）
- 解决方案：用户提供未在最近用过的全新号码池

### PayPal OTP 子链根因定位（修复实现，未经端到端验证）

#### 现象

`task_1779685723764` 完整链跑到 OTP 子链：
- ✅ paypal_approve 通过 (`ba_token` / `ec_token` / `csrf` / `nsid` 全抽到)
- ✅ OTP_INITIATE 通过（PayPal 真发 SMS，`state=PENDING`，`authId/challengeId` 拿到）
- ✅ Relay 拉到 6 位 OTP pin
- ❌ **OTP_CONFIRM 失败** `state='' first_error='PHONE_CONFIRMATION_NOT_INITIATED'`

#### 根因（HAR 实采 entry 479 / 505 反推）

PayPal weasley SDK 加载完成时会 POST 一次
`/xoplatform/logger/api/logger/`，响应 `Set-Cookie:
tsrce=checkoutuinodeweb_weasley`。这条 cookie 是 **OTP_CHALLENGE /
OTP_INITIATE / OTP_CONFIRM 三个请求被识别为合法 weasley fetch 的关键**。

协议模式直接调 OTP_CHALLENGE → 缺这条 cookie → PayPal 返回 HTML pa.js 容器
（`text/html; charset=utf-8`），客户端 JSONDecodeError 之外，**PayPal 服务端
不会建立 OTP fraud session**。OTP_INITIATE 看起来"成功"（state=PENDING + 真
发 SMS），但 OTP_CONFIRM 时 PayPal 按 fraud session 反查找不到 →
`PHONE_CONFIRMATION_NOT_INITIATED`。

#### 修复实现

- `platforms/chatgpt/paypal_http.py` 加 `paypal_post_weasley_logger`
  函数，POST `/xoplatform/logger/api/logger/`、`x-app-name=checkoutuinodeweb_weasley`，
  HAR 1:1 复刻 metric body。非阻塞契约：失败返回 False，不抛。
- `platforms/chatgpt/payment_protocol.py` `paypal_signup_otp_subchain`
  开头插入一次 `paypal_post_weasley_logger` 调用，写入 `tsrce` cookie。
- `tests/test_paypal_http.py` 加 3 个回归单测（endpoint / x-app-name /
  非阻塞契约 5xx + 网络异常）。`pytest -q` 全部通过（53 passed）。

#### 验证状态

未端到端验证：连续 2 次新任务被 PayPal 反爬阻挡：
- `task_1779686494059`：`paypal_approve` 阶段 HTTP 403 datadome challenge
  （response body 是 datadome JS challenge）
- `task_1779686964441`：Stripe `/init` curl_cffi `(35) TLS connect error`

两者都是 PayPal 服务端 IP 风控行为（datadome 频控 + 代理 IP 池被标记），
不是代码 bug。需要等冷却（建议 30 分钟以上）或换更高质量的住宅代理。
